# 使い方01 ― AIへの写像（プロンプト設計）

## このパートの考え方

本編では C を「**人がどう違うか**」を記述するために使った。
ここでは C を**逆向き**に使う。すなわち、自分や相手の C に合わせて「**出力をどう設計するか**」の設計図にする。

AIはその設計が最も効く相手だ。指示ひとつで W・n・m を直接動かせるから――人間相手より、効き目が見えやすい。

> ここからは「正しいか」ではなく「**効くか**」で見る。ツールは真である必要はなく、使えればいい。

---

## 写像表（C → プロンプト操作）

| 変数 | プロンプトでの動かし方 | 言い方の例 |
| --- | --- | --- |
| W | 焦点を I／P／O のどこに置くか指定 | 「結論だけ（O）」「根拠を添えて（I）」「考え方の筋で（P）」 |
| n_i | 与える前提・情報量 | 「この資料だけを前提に」「背景は省いて」 |
| n_o | 出力の粒度 | 「3行で」「箇条書き5点で」「網羅的に」 |
| n_p | 同時に扱う観点の数 | 「観点は3つまで」「多面的に」 |
| m | 掘り下げの段数 | 「なぜを3段掘れ」「前提を疑え」／浅く：「一段で」 |
| α・Θ・Δ | 受け手の Θ に収める | 「素人にも自動でわかる粒度で」＝相手のΘの内側へ |

---

## 同じ依頼を、相手の C で出し分ける

「新しい勤怠ルールを周知して」という一つの依頼を、宛先で変える。

- **O型の上司へ**（W=O, n_o小, m浅）
  「結論から3行。何が・いつから・現場の手間が増えるか、だけ。」
- **P型の同僚へ**（W=P, m深め）
  「なぜこの変更が必要か、旧ルールのどこが穴だったかの筋を含めて。」
- **I型の監査担当へ**（W=I, n_i大）
  「根拠規程と適用範囲、例外条件を明示して。出典つきで。」

中身は同じ。**W・n・m の指定だけ**を変えている。
（ここでの「O型／P型／I型」は相手の固定属性ではなく、その場面での志向。同じ人でも場面で変わる ― 正典1節。）

---

## 曖昧プロンプト → C明示プロンプト

> ✗「この資料、いい感じにまとめて」
> ○「この資料を前提に（n_i）、結論を先頭3行で（W=O, n_o小）、根拠は各項1行添える（I）。掘り下げは一段で（m浅）。」

「いい感じ」はAIに自分の C を推測させている。明示すれば推測のブレが消える。

---

## よくある落とし穴

- **自分の C のまま書いてしまう**：書き手が高 n・高 m だと、指定しない限りAIも高 n・高 m で返し、受け手の Θ を超える（＝「読みやすいのに読みにくい」出力）。**宛先の C を指定する。**
- **W を指定しない**：放っておくとAIは結論を急ぐO寄りに寄りやすい。筋や根拠が欲しいなら P・I・m を明示。
- **m と n_o を混同する**：「詳しく」は n_o（出力を細かく）か m（深く掘る）か曖昧。分けて指定する。

---

## クロスAI運用への接続

複数のAIを使い分けるとき、この写像表は**共通のものさし**になる。
同じ C 指定（同じ W・n・m）のプロンプトを各AIに投げ、出力差を見れば、「どのAIが P を深く掘れるか」「O要約はどれが安定か」をモデルの語彙で比較できる。
言い換え表（正典）を共有しておけば、チーム内でも「この依頼は W=P で m を2段」と短く意思疎通できる。

---

## 座標系への接続

```
IPO × C

C = (W, n, m | α, Θ, Δ)
```

本編は C を読むために、ここでは C を**書く**ために使う。同じ座標の順方向と逆方向。定義は**正典**を参照。
